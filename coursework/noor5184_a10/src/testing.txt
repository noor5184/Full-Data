Sorted:   [1, 1, 2, 3, 4, 5, 5, 6, 9]

Gnome Sort Test 5
Original: [10, 10, 3, 3, 3, 0, 100, 9, 9]
Sorted:   [0, 3, 3, 3, 9, 9, 10, 10, 100]
Swaps:    17




EXPLANATION: 

Gnome Sort has the same theoretical time complexity as both Insertion Sort and Selection Sort as in the average and worst cases all three are 0(n^2). However, its behaviour is closer to
a bubble style insertion as when it finds an out of order pair it repeatedly swaps the current element backward until it is in the correct position. This makes Gnome Sort similar to  
Insertion Sort because both can run in O(n) time on an already or almost sorted array, while Selection Sort still does Θ(n^2) comparisons even in the best case. In practice however 
Gnome Sort tends to perform more swaps than Insertion Sort on the same data, because Insertion Sort shifts values forward in the array instead of swapping on every step backward. Selection Sort 
usually minimizes swaps (about n−1) but in turn pays for that with the maximum possible number of comparisons and no advantage when the input is nearly sorted. Therefore, Gnome Sort is typically less
efficient than Insertion Sort but can outperform Selection Sort on partially sorted data, while all three remain quadratic algorithms.
